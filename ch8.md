และการจัดการลำดับชั้นหน่วยความจำ จะอธิบายได้ดีที่สุดในบริบทนี้

ได้แก่ การจัดตารางเวลาและการจัดการหน่วยความจำ

8.1 ภาพรวมของระบบปฏิบัติการ

วัตถุประสงค์และหน้าที่ของระบบปฏิบัติการ

ระบบปฏิบัติการคือโปรแกรมที่ควบคุมการทำงานของโปรแกรมประยุกต์
และทำหน้าที่เป็นส่วนต่อประสานระหว่างโปรแกรมประยุกต์กับฮาร์ดแวร์คอมพิวเตอร์
ระบบปฏิบัติการสามารถมองได้ว่ามีวัตถุประสงค์หลักสองประการ ได้แก่

■ ความสะดวกในการใช้งาน: ระบบปฏิบัติการทำให้คอมพิวเตอร์ใช้งานได้สะดวกยิ่งขึ้น
■ ประสิทธิภาพ: ระบบปฏิบัติการช่วยให้ทรัพยากรของระบบคอมพิวเตอร์ถูกใช้อย่างมีประสิทธิภาพ

ให้เราพิจารณาทั้งสองแง่มุมของระบบปฏิบัติการนี้ทีละส่วน

ระบบปฏิบัติการในฐานะส่วนต่อประสานผู้ใช้/คอมพิวเตอร์

ฮาร์ดแวร์และซอฟต์แวร์ที่ใช้ในการให้บริการโปรแกรมประยุกต์แก่ผู้ใช้
สามารถมองได้ในลักษณะลำดับชั้น ดังที่แสดงในรูปที่ 8.1
ผู้ใช้โปรแกรมประยุกต์เหล่านั้น ซึ่งก็คือผู้ใช้งานปลายทาง
โดยทั่วไปไม่ได้สนใจสถาปัตยกรรมของคอมพิวเตอร์
ดังนั้นผู้ใช้งานปลายทางจึงมองระบบคอมพิวเตอร์ในแง่ของโปรแกรมประยุกต์
โปรแกรมประยุกต์นั้นสามารถแสดงออกมาในภาษาโปรแกรมมิ่ง
และถูกพัฒนาโดยนักพัฒนาโปรแกรมประยุกต์
เพื่อพัฒนาโปรแกรมประยุกต์ในรูปแบบของชุดคำสั่งของหน่วยประมวลผล
นั้นเป็นงานที่มีความซับซ้อนอย่างมาก
เพื่อลดความซับซ้อนนี้ จึงมีการจัดเตรียมชุดของโปรแกรมระบบ
โปรแกรมบางส่วนเหล่านี้ถูกเรียกว่ายูทิลิตี
ซึ่งใช้งานฟังก์ชันที่ใช้บ่อยและช่วยเหลือในการสร้างโปรแกรม การจัดการไฟล์
และการควบคุมอุปกรณ์อินพุต/เอาต์พุต
โปรแกรมระบบที่สำคัญที่สุดคือระบบปฏิบัติการ
ระบบปฏิบัติการซ่อนรายละเอียดของฮาร์ดแวร์จากโปรแกรมเมอร์
และให้โปรแกรมเมอร์มีส่วนต่อประสานที่สะดวกสำหรับการใช้ระบบ
มันทำหน้าที่เป็นตัวกลาง ทำให้โปรแกรมเมอร์และโปรแกรมประยุกต์สามารถ
เข้าถึงและใช้งานสิ่งอำนวยความสะดวกและบริการเหล่านั้นได้ง่ายขึ้น

โดยย่อ ระบบปฏิบัติการมักให้บริการในพื้นที่ต่อไปนี้

■ การสร้างโปรแกรม:
ระบบปฏิบัติการจัดเตรียมสิ่งอำนวยความสะดวกและบริการต่าง ๆ มากมาย
เช่น โปรแกรมแก้ไข (editor) และโปรแกรมดีบัก (debugger)
เพื่อช่วยเหลือโปรแกรมเมอร์ในการสร้างโปรแกรม
โดยทั่วไปบริการเหล่านี้อยู่ในรูปแบบของโปรแกรมยูทิลิตี
ที่ไม่ได้เป็นส่วนหนึ่งของระบบปฏิบัติการจริง ๆ
แต่สามารถเข้าถึงได้ผ่านระบบปฏิบัติการ

■ การรันโปรแกรม:
การรันโปรแกรมต้องผ่านหลายขั้นตอน
คำสั่งและข้อมูลต้องถูกโหลดเข้าสู่หน่วยความจำหลัก
อุปกรณ์อินพุต/เอาต์พุตและไฟล์ต้องได้รับการเตรียมพร้อม
และทรัพยากรอื่น ๆ ต้องถูกจัดเตรียม
ระบบปฏิบัติการจัดการทุกอย่างเหล่านี้แทนผู้ใช้

■ การเข้าถึงอุปกรณ์อินพุต/เอาต์พุต:
อุปกรณ์อินพุต/เอาต์พุตแต่ละชนิดต้องการชุดคำสั่งหรือสัญญาณควบคุมเฉพาะของตนสำหรับการทำงาน
ระบบปฏิบัติการจัดการรายละเอียดเหล่านี้
เพื่อให้โปรแกรมเมอร์สามารถคิดในแง่ของการอ่านและเขียนอย่างง่ายได้

■ การควบคุมการเข้าถึงไฟล์:
ในกรณีของไฟล์ การควบคุมต้องรวมถึงความเข้าใจ
ไม่เพียงแต่ลักษณะของอุปกรณ์อินพุต/เอาต์พุต (ดิสก์ไดรฟ์ เทปไดรฟ์)
แต่ยังรวมถึงรูปแบบไฟล์บนสื่อบันทึกข้อมูลด้วย
ระบบปฏิบัติการจัดการรายละเอียดเหล่านี้
นอกจากนี้ในกรณีของระบบที่มีผู้ใช้หลายคนพร้อมกัน
ระบบปฏิบัติการสามารถให้กลไกการป้องกันเพื่อควบคุมการเข้าถึงไฟล์ได้

■ การเข้าถึงระบบ:
ในกรณีของระบบที่ใช้ร่วมกันหรือระบบสาธารณะ
ระบบปฏิบัติการควบคุมการเข้าถึงระบบโดยรวมและทรัพยากรระบบเฉพาะต่าง ๆ
ฟังก์ชันการเข้าถึงต้องให้การป้องกันทรัพยากรและข้อมูลจากผู้ใช้ที่ไม่ได้รับอนุญาต
และต้องแก้ไขข้อขัดแย้งในการแย่งชิงทรัพยากร

■ การตรวจจับข้อผิดพลาดและการตอบสนอง:
ข้อผิดพลาดหลากหลายประเภทอาจเกิดขึ้นขณะระบบคอมพิวเตอร์กำลังทำงาน
ซึ่งรวมถึงข้อผิดพลาดของฮาร์ดแวร์ทั้งภายในและภายนอก
เช่น ข้อผิดพลาดของหน่วยความจำ หรืออุปกรณ์ขัดข้อง
และข้อผิดพลาดของซอฟต์แวร์ต่าง ๆ
เช่น การล้นค่าทางคณิตศาสตร์
การพยายามเข้าถึงพื้นที่หน่วยความจำที่ต้องห้าม
และการที่ระบบปฏิบัติการไม่สามารถตอบสนองคำขอของโปรแกรมประยุกต์ได้
ในแต่ละกรณี ระบบปฏิบัติการต้องตอบสนองด้วยวิธีที่ล้างสภาวะข้อผิดพลาด
โดยกระทบต่อโปรแกรมที่กำลังทำงานอยู่น้อยที่สุด
การตอบสนองอาจมีตั้งแต่การยุติโปรแกรมที่ทำให้เกิดข้อผิดพลาด
ไปจนถึงการลองดำเนินการใหม่
หรือเพียงแค่รายงานข้อผิดพลาดให้โปรแกรมประยุกต์ทราบ

■ การบัญชี:
ระบบปฏิบัติการที่ดีจะเก็บสถิติการใช้งานสำหรับทรัพยากรต่าง ๆ
และตรวจสอบพารามิเตอร์ด้านประสิทธิภาพ เช่น เวลาในการตอบสนอง
ในระบบใด ๆ ข้อมูลนี้มีประโยชน์ในการคาดการณ์ความต้องการการปรับปรุงในอนาคต
และในการปรับแต่งระบบเพื่อปรับปรุงประสิทธิภาพ
ในระบบที่มีผู้ใช้หลายคน ข้อมูลนี้สามารถใช้เพื่อวัตถุประสงค์ด้านการเรียกเก็บเงินได้

รูปที่ 8.1 ยังระบุส่วนต่อประสานหลักสามประการในระบบคอมพิวเตอร์ทั่วไป

■ สถาปัตยกรรมชุดคำสั่ง (Instruction Set Architecture: ISA):
ISA กำหนดคลังของคำสั่งภาษาเครื่องที่คอมพิวเตอร์สามารถปฏิบัติตามได้
ส่วนต่อประสานนี้คือขอบเขตระหว่างฮาร์ดแวร์และซอฟต์แวร์
โปรดสังเกตว่าทั้งโปรแกรมประยุกต์และยูทิลิตีอาจเข้าถึง ISA โดยตรงได้
สำหรับโปรแกรมเหล่านี้ มีชุดย่อยของคลังคำสั่งให้ใช้ (user ISA)
ระบบปฏิบัติการมีสิทธิ์เข้าถึงคำสั่งภาษาเครื่องเพิ่มเติม
ที่เกี่ยวข้องกับการจัดการทรัพยากรระบบ (system ISA)

■ ส่วนต่อประสานไบนารีโปรแกรมประยุกต์ (Application Binary Interface: ABI):
ABI กำหนดมาตรฐานสำหรับความสามารถในการพกพาไบนารีระหว่างโปรแกรม
ABI กำหนดส่วนต่อประสานการเรียกระบบไปยังระบบปฏิบัติการ
และทรัพยากรและบริการของฮาร์ดแวร์ที่มีอยู่ในระบบผ่าน user ISA

■ ส่วนต่อประสานการโปรแกรมประยุกต์ (Application Programming Interface: API):
API ให้โปรแกรมสามารถเข้าถึงทรัพยากรและบริการของฮาร์ดแวร์
ที่มีอยู่ในระบบผ่าน user ISA เสริมด้วยการเรียกไลบรารีภาษาระดับสูง (HLL)
การเรียกระบบใด ๆ มักดำเนินการผ่านไลบรารี
การใช้ API ช่วยให้ซอฟต์แวร์ประยุกต์สามารถพกพาได้ง่าย
ผ่านการคอมไพล์ใหม่ ไปยังระบบอื่นที่รองรับ API เดียวกัน

ระบบปฏิบัติการในฐานะผู้จัดการทรัพยากร

คอมพิวเตอร์คือชุดของทรัพยากรสำหรับการเคลื่อนย้าย จัดเก็บ และประมวลผลข้อมูล
และสำหรับการควบคุมฟังก์ชันเหล่านี้
ระบบปฏิบัติการมีหน้าที่จัดการทรัพยากรเหล่านี้

เราสามารถกล่าวได้ว่าระบบปฏิบัติการควบคุมการเคลื่อนย้าย การจัดเก็บ และการประมวลผลข้อมูลได้หรือไม่
จากมุมมองหนึ่ง คำตอบคือใช่
โดยการจัดการทรัพยากรของคอมพิวเตอร์ ระบบปฏิบัติการอยู่ในการควบคุมฟังก์ชันพื้นฐานของคอมพิวเตอร์
แต่การควบคุมนี้ใช้งานในลักษณะที่น่าสนใจ

ระบบปฏิบัติการมีลักษณะพิเศษในฐานะกลไกควบคุมในสองด้าน

■ ระบบปฏิบัติการทำงานในลักษณะเดียวกับซอฟต์แวร์คอมพิวเตอร์ทั่วไป
กล่าวคือมันเป็นโปรแกรมที่ถูกประมวลผลโดยหน่วยประมวลผล

■ ระบบปฏิบัติการสละการควบคุมอยู่บ่อยครั้ง
และต้องพึ่งพาหน่วยประมวลผลเพื่อให้ได้การควบคุมกลับคืน

เช่นเดียวกับโปรแกรมคอมพิวเตอร์อื่น ๆ ระบบปฏิบัติการให้คำสั่งสำหรับหน่วยประมวลผล
ความแตกต่างที่สำคัญอยู่ที่จุดประสงค์ของโปรแกรม
ระบบปฏิบัติการชี้นำหน่วยประมวลผลในการใช้ทรัพยากรระบบอื่น ๆ
และในการกำหนดเวลาการประมวลผลโปรแกรมอื่น ๆ
แต่เพื่อให้หน่วยประมวลผลทำสิ่งเหล่านั้นได้
มันต้องหยุดประมวลผลโปรแกรมระบบปฏิบัติการและประมวลผลโปรแกรมอื่นแทน
ดังนั้นระบบปฏิบัติการจึงสละการควบคุมเพื่อให้หน่วยประมวลผลทำงาน "ที่มีประโยชน์"
และจากนั้นจึงกลับมาควบคุมนานพอที่จะเตรียมหน่วยประมวลผลสำหรับงานชิ้นต่อไป

รูปที่ 8.2 แสดงให้เห็นทรัพยากรหลักที่ระบบปฏิบัติการจัดการ
ส่วนหนึ่งของระบบปฏิบัติการอยู่ในหน่วยความจำหลัก
ซึ่งรวมถึงเคอร์เนล (kernel) หรือนิวเคลียส (nucleus)
ซึ่งมีฟังก์ชันที่ใช้บ่อยที่สุดในระบบปฏิบัติการ
ส่วนที่เหลือของหน่วยความจำหลักประกอบด้วยโปรแกรมและข้อมูลของผู้ใช้

ประเภทของระบบปฏิบัติการ

ลักษณะสำคัญบางประการใช้เพื่อจำแนกระบบปฏิบัติการประเภทต่าง ๆ
ลักษณะเหล่านี้อยู่ใน 2 มิติอิสระ
มิติแรกระบุว่าระบบเป็นแบบแบตช์หรือแบบโต้ตอบ

ในระบบโต้ตอบ ผู้ใช้/โปรแกรมเมอร์โต้ตอบโดยตรงกับคอมพิวเตอร์
โดยปกติผ่านเทอร์มินัลแป้นพิมพ์/จอภาพ
เพื่อร้องขอการรันงานหรือดำเนินธุรกรรม
ยิ่งกว่านั้น ผู้ใช้อาจสื่อสารกับคอมพิวเตอร์ระหว่างการรันงาน
ขึ้นอยู่กับลักษณะของโปรแกรมประยุกต์

ระบบแบตช์เป็นสิ่งตรงข้ามกับระบบโต้ตอบ
โปรแกรมของผู้ใช้ถูกรวมเข้ากับโปรแกรมจากผู้ใช้คนอื่น
และส่งโดยผู้ดูแลระบบคอมพิวเตอร์
หลังจากโปรแกรมเสร็จสิ้น ผลลัพธ์จะถูกพิมพ์ออกมาให้ผู้ใช้
ระบบแบตช์แบบล้วน ๆ หายากในปัจจุบัน
อย่างไรก็ตามจะเป็นประโยชน์สำหรับการอธิบายระบบปฏิบัติการสมัยใหม่
ที่จะตรวจสอบระบบแบตช์โดยสังเขป

มิติอิสระระบุว่าระบบใช้มัลติโปรแกรมมิ่งหรือไม่
ด้วยมัลติโปรแกรมมิ่ง ความพยายามเกิดขึ้นเพื่อให้หน่วยประมวลผลทำงานอยู่เสมอ
โดยให้มันทำงานกับโปรแกรมมากกว่าหนึ่งโปรแกรมในคราวเดียว
โปรแกรมหลายโปรแกรมถูกโหลดเข้าสู่หน่วยความจำ
และหน่วยประมวลผลสลับระหว่างโปรแกรมเหล่านั้นอย่างรวดเร็ว
ทางเลือกอื่นคือระบบยูนิโปรแกรมมิ่งที่ทำงานกับโปรแกรมเดียวในแต่ละครั้ง

ระบบยุคแรก

ในคอมพิวเตอร์ยุคแรกสุด ตั้งแต่ปลายทศวรรษ 1940 ถึงกลางทศวรรษ 1950
โปรแกรมเมอร์โต้ตอบโดยตรงกับฮาร์ดแวร์คอมพิวเตอร์โดยไม่มีระบบปฏิบัติการ
หน่วยประมวลผลเหล่านี้ถูกควบคุมจากคอนโซล
ซึ่งประกอบด้วยไฟแสดงสถานะ สวิตช์สลับ อุปกรณ์อินพุตบางรูปแบบ และเครื่องพิมพ์
โปรแกรมในรหัสหน่วยประมวลผลถูกโหลดผ่านอุปกรณ์อินพุต (เช่น เครื่องอ่านการ์ด)
หากข้อผิดพลาดหยุดโปรแกรม สภาวะข้อผิดพลาดจะถูกระบุด้วยไฟ
โปรแกรมเมอร์สามารถตรวจสอบรีจิสเตอร์และหน่วยความจำหลัก
เพื่อหาสาเหตุของข้อผิดพลาดได้
หากโปรแกรมดำเนินไปจนเสร็จสมบูรณ์ตามปกติ ผลลัพธ์จะปรากฏบนเครื่องพิมพ์

ระบบยุคแรกเหล่านี้มีปัญหาหลักสองประการ

■ การจัดตาราง:
สถานติดตั้งส่วนใหญ่ใช้แผ่นลงชื่อเพื่อจองเวลาหน่วยประมวลผล
โดยทั่วไปผู้ใช้สามารถลงชื่อจองเวลาเป็นชั่วโมงในหน่วยครึ่งชั่วโมงหรือมากกว่า
ผู้ใช้อาจลงชื่อจองหนึ่งชั่วโมงและเสร็จใน 45 นาที
ส่งผลให้คอมพิวเตอร์ว่างงานสูญเปล่า
ในทางกลับกัน ผู้ใช้อาจพบปัญหา ไม่เสร็จในเวลาที่จัดสรร
และถูกบังคับให้หยุดก่อนที่จะแก้ปัญหาได้

■ เวลาในการเตรียมงาน:
โปรแกรมเดียวที่เรียกว่างาน (job) อาจเกี่ยวข้องกับ
การโหลดคอมไพเลอร์บวกกับโปรแกรมภาษาระดับสูง (โปรแกรมต้นฉบับ) เข้าสู่หน่วยความจำ
การบันทึกโปรแกรมที่คอมไพล์แล้ว (โปรแกรมออบเจ็กต์)
และจากนั้นการโหลดและเชื่อมโยงโปรแกรมออบเจ็กต์และฟังก์ชันทั่วไปเข้าด้วยกัน
แต่ละขั้นตอนเหล่านี้อาจเกี่ยวข้องกับการติดตั้งหรือถอดเทป หรือการตั้งค่าชุดการ์ด
หากเกิดข้อผิดพลาด ผู้ใช้ที่โชคร้ายมักต้องกลับไปเริ่มต้นลำดับการตั้งค่าใหม่
ดังนั้นจึงใช้เวลามากในการตั้งค่าโปรแกรมเพื่อรัน

8.1 / ภาพรวมของระบบปฏิบัติการ 278

โหมดการทำงานนี้สามารถเรียกว่าการประมวลผลแบบอนุกรม (serial processing)
สะท้อนความจริงที่ว่าผู้ใช้เข้าถึงคอมพิวเตอร์ตามลำดับ
เมื่อเวลาผ่านไป ซอฟต์แวร์ระบบต่าง ๆ ได้รับการพัฒนา
เพื่อพยายามทำให้การประมวลผลแบบอนุกรมมีประสิทธิภาพมากขึ้น
ซึ่งรวมถึงไลบรารีของฟังก์ชันทั่วไป ลิงเกอร์ โหลดเดอร์ ดีบักเกอร์
และรูทีนไดรเวอร์อินพุต/เอาต์พุต
ที่มีให้ใช้เป็นซอฟต์แวร์ทั่วไปสำหรับผู้ใช้ทุกคน

ระบบแบตช์อย่างง่าย

หน่วยประมวลผลยุคแรกมีราคาแพงมาก
ดังนั้นจึงสำคัญที่จะต้องเพิ่มการใช้ประโยชน์สูงสุด
เวลาที่สูญเปล่าเนื่องจากการจัดตารางและเวลาในการเตรียมงานนั้นเป็นสิ่งที่ยอมรับไม่ได้

เพื่อปรับปรุงการใช้ประโยชน์ จึงได้พัฒนาระบบปฏิบัติการแบบแบตช์อย่างง่ายขึ้น
ด้วยระบบดังกล่าว ที่เรียกว่า monitor
ผู้ใช้ไม่มีสิทธิ์เข้าถึงหน่วยประมวลผลโดยตรงอีกต่อไป
แต่ผู้ใช้ส่งงานบนการ์ดหรือเทปให้ผู้ดูแลระบบคอมพิวเตอร์
ซึ่งรวบรวมงานตามลำดับและวางชุดงานทั้งหมดบนอุปกรณ์อินพุตเพื่อให้ monitor ใช้งาน

เพื่อให้เข้าใจว่าระบบนี้ทำงานอย่างไร
ให้เราดูจากสองมุมมอง คือ มุมมองของ monitor และมุมมองของหน่วยประมวลผล
จากมุมมองของ monitor นั้น monitor ควบคุมลำดับของเหตุการณ์
เพื่อให้เป็นเช่นนั้น ส่วนใหญ่ของ monitor ต้องอยู่ในหน่วยความจำหลักเสมอ
และพร้อมสำหรับการประมวลผล (รูปที่ 8.3)
ส่วนนั้นเรียกว่า resident monitor
ส่วนที่เหลือของ monitor ประกอบด้วยยูทิลิตีและฟังก์ชันทั่วไป
ที่ถูกโหลดเป็นซับรูทีนสำหรับโปรแกรมผู้ใช้ที่จุดเริ่มต้นของงานใด ๆ ที่ต้องการ
monitor อ่านงานครั้งละหนึ่งงานจากอุปกรณ์อินพุต
(โดยทั่วไปคือเครื่องอ่านการ์ดหรือไดรฟ์เทปแม่เหล็ก)
ขณะที่อ่าน งานปัจจุบันจะถูกวางในพื้นที่โปรแกรมผู้ใช้
และการควบคุมถูกส่งต่อไปยังงานนี้
เมื่องานเสร็จสมบูรณ์ มันส่งคืนการควบคุมไปยัง monitor
ซึ่งอ่านงานถัดไปทันที
ผลลัพธ์ของแต่ละงานจะถูกพิมพ์ออกมาเพื่อส่งมอบให้ผู้ใช้

279 บทที่ 8 / การสนับสนุนของระบบปฏิบัติการ

ตอนนี้ลองพิจารณาลำดับนี้จากมุมมองของหน่วยประมวลผล
ณ จุดหนึ่งในเวลา หน่วยประมวลผลกำลังประมวลผลคำสั่ง
จากส่วนของหน่วยความจำหลักที่มี monitor อยู่
คำสั่งเหล่านี้ทำให้งานถัดไปถูกอ่านเข้าไปยังส่วนอื่นของหน่วยความจำหลัก
เมื่องานถูกอ่านเข้ามาแล้ว หน่วยประมวลผลจะพบคำสั่ง branch ใน monitor
ที่สั่งให้หน่วยประมวลผลดำเนินการต่อที่จุดเริ่มต้นของโปรแกรมผู้ใช้
หน่วยประมวลผลจะประมวลผลคำสั่งในโปรแกรมของผู้ใช้
จนกว่าจะพบสภาวะการสิ้นสุดหรือข้อผิดพลาด
ทั้งสองเหตุการณ์นี้ทำให้หน่วยประมวลผลดึงคำสั่งถัดไปจากโปรแกรม monitor

ควรชัดเจนว่า monitor จัดการปัญหาการจัดตาราง
ชุดงานถูกจัดคิวไว้ และงานถูกประมวลผลอย่างรวดเร็วที่สุดโดยไม่มีเวลาว่างระหว่างงาน

เกี่ยวกับเวลาในการเตรียมงาน monitor จัดการสิ่งนี้เช่นกัน
ด้วยแต่ละงาน คำสั่งจะรวมอยู่ใน job control language (JCL)
ซึ่งเป็นภาษาโปรแกรมพิเศษที่ใช้เพื่อให้คำสั่งแก่ monitor

monitor หรือระบบปฏิบัติการแบบแบตช์เป็นเพียงโปรแกรมคอมพิวเตอร์
มันพึ่งพาความสามารถของหน่วยประมวลผลในการดึงคำสั่งจากส่วนต่าง ๆ
ของหน่วยความจำหลักเพื่อยึดและสละการควบคุมสลับกัน
คุณสมบัติฮาร์ดแวร์บางอย่างอื่น ๆ ยังเป็นที่ต้องการด้วย

■ การป้องกันหน่วยความจำ (Memory Protection):
ในขณะที่โปรแกรมผู้ใช้กำลังประมวลผล
ต้องไม่แก้ไขพื้นที่หน่วยความจำที่มี monitor อยู่
หากมีการพยายามดังกล่าว ฮาร์ดแวร์หน่วยประมวลผลควรตรวจจับข้อผิดพลาด
และโอนการควบคุมไปยัง monitor
monitor จากนั้นจะยกเลิกงาน พิมพ์ข้อความแสดงข้อผิดพลาด และโหลดงานถัดไป

■ ตัวจับเวลา (Timer):
ตัวจับเวลาใช้เพื่อป้องกันไม่ให้งานเดียวครอบครองระบบ
ตัวจับเวลาถูกตั้งค่าที่จุดเริ่มต้นของแต่ละงาน
หากตัวจับเวลาหมดอายุ จะเกิดการขัดจังหวะและการควบคุมกลับไปยัง monitor

■ คำสั่งสิทธิ์พิเศษ (Privileged Instructions):
คำสั่งบางอย่างถูกกำหนดให้เป็นสิทธิ์พิเศษ
และสามารถดำเนินการได้โดย monitor เท่านั้น
หากหน่วยประมวลผลพบคำสั่งดังกล่าวขณะประมวลผลโปรแกรมผู้ใช้
จะเกิดการขัดจังหวะข้อผิดพลาด
ในบรรดาคำสั่งสิทธิ์พิเศษ ได้แก่ คำสั่งอินพุต/เอาต์พุต
เพื่อให้ monitor ยังคงควบคุมอุปกรณ์อินพุต/เอาต์พุตทั้งหมด
หากโปรแกรมผู้ใช้ต้องการดำเนินการอินพุต/เอาต์พุต
ต้องร้องขอให้ monitor ดำเนินการแทน

■ การขัดจังหวะ (Interrupts):
คอมพิวเตอร์รุ่นแรกไม่มีความสามารถนี้
คุณสมบัตินี้ให้ระบบปฏิบัติการมีความยืดหยุ่นมากขึ้น
ในการสละและกลับมาควบคุมจากโปรแกรมผู้ใช้

เวลาหน่วยประมวลผลสลับระหว่างการประมวลผลโปรแกรมผู้ใช้
และการประมวลผล monitor
มีการเสียสละสองอย่างเกิดขึ้น ได้แก่
หน่วยความจำหลักบางส่วนถูกมอบให้ monitor
และเวลาหน่วยประมวลผลบางส่วนถูกใช้โดย monitor
แม้จะมีค่าโสหุ้ยนี้ ระบบแบตช์อย่างง่ายก็ปรับปรุงการใช้ประโยชน์ของคอมพิวเตอร์

ระบบแบตช์แบบมัลติโปรแกรมมิ่ง

แม้แต่ในการจัดลำดับงานอัตโนมัติที่จัดเตรียมโดยระบบปฏิบัติการแบบแบตช์อย่างง่าย
หน่วยประมวลผลก็มักว่างงาน
ปัญหาคืออุปกรณ์อินพุต/เอาต์พุตช้ากว่าหน่วยประมวลผลมาก
รูปที่ 8.4 แสดงรายละเอียดการคำนวณตัวแทน
การคำนวณเกี่ยวข้องกับโปรแกรมที่ประมวลผลไฟล์ระเบียน
และดำเนินการ เฉลี่ย 100 คำสั่งหน่วยประมวลผลต่อระเบียน
ในตัวอย่างนี้คอมพิวเตอร์ใช้เวลากว่า 96%
ในการรออุปกรณ์อินพุต/เอาต์พุตให้เสร็จสิ้นการถ่ายโอนข้อมูล
รูปที่ 8.5a แสดงสถานการณ์นี้

ความไม่มีประสิทธิภาพนี้ไม่จำเป็น
เรารู้ว่าต้องมีหน่วยความจำเพียงพอสำหรับระบบปฏิบัติการ (resident monitor)
และโปรแกรมผู้ใช้หนึ่งโปรแกรม
สมมติว่ามีพื้นที่สำหรับระบบปฏิบัติการและโปรแกรมผู้ใช้สองโปรแกรม
ตอนนี้เมื่องานหนึ่งต้องรออินพุต/เอาต์พุต
หน่วยประมวลผลสามารถสลับไปยังอีกงานหนึ่ง
ซึ่งน่าจะไม่ได้รออินพุต/เอาต์พุต (รูปที่ 8.5b)
นอกจากนี้เราอาจขยายหน่วยความจำเพื่อรองรับสาม สี่ หรือมากกว่านั้น
และสลับระหว่างโปรแกรมทั้งหมด (รูปที่ 8.5c)
เทคนิคนี้รู้จักกันในชื่อมัลติโปรแกรมมิ่ง (multiprogramming)
หรือมัลติแทสกิ้ง (multitasking)
ซึ่งเป็นหัวข้อหลักของระบบปฏิบัติการสมัยใหม่

8.2 การจัดตารางเวลา (Scheduling)

ตารางที่ 8.3 แสดงให้เห็นชนิดของการจัดตารางเวลาที่ระบบปฏิบัติการดำเนินการ

การจัดตารางเวลาระยะยาว (Long-term Scheduling):
การตัดสินใจเพิ่มจำนวนกระบวนการที่อยู่ในหน่วยความจำหลักบางส่วนหรือทั้งหมด

การจัดตารางเวลาระยะกลาง (Medium-term Scheduling):
การตัดสินใจเพิ่มจำนวนกระบวนการที่กำลังทำงานอยู่อย่างแข็งขัน

การจัดตารางเวลาระยะสั้น (Short-term Scheduling):
การตัดสินใจว่ากระบวนการใดที่พร้อมจะถูกประมวลผลโดยหน่วยประมวลผลต่อไป

การจัดตารางเวลาอินพุต/เอาต์พุต (I/O Scheduling):
การตัดสินใจว่าคำขออินพุต/เอาต์พุตที่ค้างอยู่ของกระบวนการใด
จะถูกจัดการโดยอุปกรณ์อินพุต/เอาต์พุตที่พร้อมใช้งาน

การจัดตารางเวลาระยะยาว (Long-term Scheduling)

ในระบบแบตช์ หรือสำหรับส่วนแบตช์ของระบบปฏิบัติการทั่วไป
งานที่ส่งเข้ามาใหม่จะถูกส่งไปยังดิสก์และเก็บไว้ในคิวแบตช์
ตัวกำหนดตารางเวลาระยะยาวสร้างกระบวนการจากคิวเมื่อสามารถทำได้
มีการตัดสินใจสองอย่างที่เกี่ยวข้อง
ประการแรก ตัวกำหนดตารางเวลาต้องตัดสินใจว่า
ระบบปฏิบัติการสามารถรับกระบวนการเพิ่มเติมหนึ่งหรือมากกว่าได้หรือไม่
ประการที่สอง ตัวกำหนดตารางเวลาต้องตัดสินใจว่างานหรืองานใดที่จะยอมรับ
และเปลี่ยนเป็นกระบวนการ
เกณฑ์ที่ใช้อาจรวมถึงลำดับความสำคัญ เวลาในการประมวลผลที่คาดการณ์ไว้
และความต้องการอินพุต/เอาต์พุต

สำหรับโปรแกรมโต้ตอบในระบบแบบแบ่งเวลา
คำขอกระบวนการจะถูกสร้างขึ้นเมื่อผู้ใช้พยายามเชื่อมต่อกับระบบ
ผู้ใช้แบบแบ่งเวลาไม่ได้ถูกจัดคิวและรอจนกว่าระบบจะรับได้
แต่ระบบปฏิบัติการจะรับผู้ใช้ที่ได้รับอนุญาตทั้งหมดจนกว่าระบบจะอิ่มตัว
โดยใช้การวัดความอิ่มตัวที่กำหนดไว้ล่วงหน้า
ณ จุดนั้น คำขอการเชื่อมต่อจะได้รับข้อความแจ้งว่าระบบเต็ม
และผู้ใช้ควรลองอีกครั้งในภายหลัง

การจัดตารางเวลาระยะกลาง (Medium-term Scheduling)

การจัดตารางเวลาระยะกลางเป็นส่วนหนึ่งของฟังก์ชันการสวอปปิ้ง
ซึ่งอธิบายไว้ในหัวข้อ 8.3
โดยทั่วไป การตัดสินใจสวอปอิน (swapping-in)
จะอ้างอิงจากความจำเป็นในการจัดการระดับของมัลติโปรแกรมมิ่ง
ในระบบที่ไม่ใช้หน่วยความจำเสมือน การจัดการหน่วยความจำก็เป็นประเด็นเช่นกัน
ดังนั้นการตัดสินใจสวอปอินจะพิจารณาความต้องการหน่วยความจำ
ของกระบวนการที่ถูกสวอปออก

การจัดตารางเวลาระยะสั้น (Short-term Scheduling)

ตัวกำหนดตารางเวลาระยะยาวทำงานค่อนข้างไม่บ่อยนัก
และตัดสินใจแบบหยาบ ๆ ว่าจะรับกระบวนการใหม่หรือไม่ และจะรับกระบวนการใด
ตัวกำหนดตารางเวลาระยะสั้น ที่รู้จักกันในชื่อ dispatcher
ทำงานบ่อยครั้งและตัดสินใจในรายละเอียดว่างานใดจะถูกประมวลผลต่อไป

สถานะของกระบวนการ (Process States)

เพื่อให้เข้าใจการทำงานของตัวกำหนดตารางเวลาระยะสั้น
เราต้องพิจารณาแนวคิดของสถานะกระบวนการ
ตลอดอายุของกระบวนการ สถานะของมันจะเปลี่ยนแปลงหลายครั้ง
สถานะของมัน ณ จุดเวลาใดก็ตามเรียกว่า state
คำว่า state ถูกใช้เพราะมันบ่งบอกว่ามีข้อมูลบางอย่างที่กำหนดสถานะ ณ จุดนั้น
อย่างน้อยมีห้าสถานะที่กำหนดไว้สำหรับกระบวนการ (รูปที่ 8.7)

■ New:
โปรแกรมได้รับการอนุมัติโดย high-level scheduler
แต่ยังไม่พร้อมสำหรับการประมวลผล
ระบบปฏิบัติการจะเริ่มต้นกระบวนการ โดยย้ายไปยังสถานะ ready

■ Ready:
กระบวนการพร้อมสำหรับการประมวลผล
และรอการเข้าถึงหน่วยประมวลผล

■ Running:
กระบวนการกำลังถูกประมวลผลโดยหน่วยประมวลผล

■ Waiting:
กระบวนการถูกระงับจากการประมวลผลในขณะที่รอทรัพยากรระบบบางอย่าง
เช่น อินพุต/เอาต์พุต

■ Halted:
กระบวนการสิ้นสุดและจะถูกทำลายโดยระบบปฏิบัติการ

สำหรับแต่ละกระบวนการในระบบ ระบบปฏิบัติการต้องเก็บรักษาข้อมูล
ที่ระบุสถานะของกระบวนการและข้อมูลอื่น ๆ ที่จำเป็นสำหรับการประมวลผลกระบวนการ
เพื่อวัตถุประสงค์นี้ แต่ละกระบวนการจะถูกแทนด้วย
process control block (รูปที่ 8.8) ซึ่งมักประกอบด้วย

■ Identifier: กระบวนการที่กำลังทำงานแต่ละกระบวนการมีตัวระบุเฉพาะ

■ State: สถานะปัจจุบันของกระบวนการ (new, ready และอื่น ๆ)

■ Priority: ระดับความสำคัญสัมพัทธ์

■ Program counter:
ที่อยู่ของคำสั่งถัดไปในโปรแกรมที่จะถูกประมวลผล

■ Memory pointers:
ตำแหน่งเริ่มต้นและสิ้นสุดของกระบวนการในหน่วยความจำ

■ Context data:
เหล่านี้คือข้อมูลที่อยู่ในรีจิสเตอร์ในหน่วยประมวลผลขณะกระบวนการกำลังประมวลผล
Context data พร้อมกับ program counter
จะถูกบันทึกเมื่อกระบวนการออกจากสถานะ running
พวกมันถูกดึงกลับโดยหน่วยประมวลผลเมื่อมันกลับมาประมวลผลกระบวนการ

■ I/O status information:
รวมถึงคำขออินพุต/เอาต์พุตที่ค้างอยู่
อุปกรณ์อินพุต/เอาต์พุต (เช่น เทปไดรฟ์) ที่กำหนดให้กับกระบวนการนี้
รายการไฟล์ที่กำหนดให้กับกระบวนการ และอื่น ๆ

■ Accounting information:
อาจรวมถึงปริมาณเวลาหน่วยประมวลผลและเวลานาฬิกาที่ใช้
ขีดจำกัดเวลา หมายเลขบัญชี และอื่น ๆ

เมื่อตัวกำหนดตารางเวลารับงานใหม่หรือคำขอผู้ใช้สำหรับการประมวลผล
มันจะสร้าง process control block ว่าง
และวางกระบวนการที่เกี่ยวข้องในสถานะ new
หลังจากระบบได้กรอกข้อมูล process control block อย่างถูกต้องแล้ว
กระบวนการจะถูกโอนไปยังสถานะ ready

เทคนิคการจัดตารางเวลา (Scheduling Techniques)

เพื่อให้เข้าใจว่าระบบปฏิบัติการจัดการการจัดตารางเวลาของงานต่าง ๆ
ในหน่วยความจำอย่างไร
ให้เราเริ่มต้นโดยพิจารณาตัวอย่างง่าย ๆ ในรูปที่ 8.9
รูปแสดงให้เห็นว่าหน่วยความจำหลักถูกแบ่งพาร์ติชันอย่างไร ณ จุดเวลาหนึ่ง
เคอร์เนลของระบบปฏิบัติการอยู่ที่นั่นเสมอ
นอกจากนี้ยังมีกระบวนการที่ทำงานอยู่จำนวนหนึ่ง รวมถึง A และ B
ซึ่งแต่ละกระบวนการได้รับการจัดสรรส่วนหนึ่งของหน่วยความจำ

เราเริ่มต้น ณ จุดเวลาที่กระบวนการ A กำลังทำงาน
หน่วยประมวลผลกำลังประมวลผลคำสั่งจากโปรแกรมที่อยู่ในพาร์ติชันหน่วยความจำของ A
ณ จุดเวลาต่อมา หน่วยประมวลผลหยุดประมวลผลคำสั่งใน A
และเริ่มประมวลผลคำสั่งในพื้นที่ระบบปฏิบัติการ
สิ่งนี้จะเกิดขึ้นด้วยเหตุผลหนึ่งในสามเหตุผล

1. กระบวนการ A ส่งการเรียกบริการ (service call)
เช่น คำขออินพุต/เอาต์พุต ไปยังระบบปฏิบัติการ
การประมวลผลของ A ถูกระงับจนกว่าการเรียกนี้จะได้รับการตอบสนอง
โดยระบบปฏิบัติการ

2. กระบวนการ A ทำให้เกิดการขัดจังหวะ (interrupt)
การขัดจังหวะคือสัญญาณที่สร้างโดยฮาร์ดแวร์ไปยังหน่วยประมวลผล
เมื่อตรวจจับสัญญาณนี้ได้
หน่วยประมวลผลหยุดประมวลผล A
และโอนไปยัง interrupt handler ในระบบปฏิบัติการ
เหตุการณ์ต่าง ๆ ที่เกี่ยวข้องกับ A จะทำให้เกิดการขัดจังหวะ
ตัวอย่างหนึ่งคือข้อผิดพลาด เช่น ความพยายามประมวลผลคำสั่งสิทธิ์พิเศษ
อีกตัวอย่างหนึ่งคือการหมดเวลา
เพื่อป้องกันไม่ให้กระบวนการใดครอบครองหน่วยประมวลผล
แต่ละกระบวนการจะได้รับหน่วยประมวลผลเพียงช่วงเวลาสั้น ๆ ในแต่ละครั้ง

3. เหตุการณ์บางอย่างที่ไม่เกี่ยวข้องกับกระบวนการ A
ที่ต้องการความสนใจทำให้เกิดการขัดจังหวะ
ตัวอย่างคือการเสร็จสิ้นของการดำเนินการอินพุต/เอาต์พุต

ในกรณีใดก็ตาม ผลลัพธ์คือดังต่อไปนี้
หน่วยประมวลผลบันทึก context data ปัจจุบัน
และ program counter ของ A ใน process control block ของ A
จากนั้นเริ่มประมวลผลใน OS
OS อาจดำเนินการบางอย่าง เช่น การเริ่มต้นการดำเนินการอินพุต/เอาต์พุต
จากนั้นส่วน short-term-scheduler ของ OS
ตัดสินใจว่ากระบวนการใดควรถูกประมวลผลต่อไป
ในตัวอย่างนี้ B ถูกเลือก
OS สั่งให้หน่วยประมวลผลคืนค่า context data ของ B
และดำเนินการประมวลผล B ต่อจากที่หยุดไว้

ตัวอย่างง่าย ๆ นี้เน้นการทำงานพื้นฐานของตัวกำหนดตารางเวลาระยะสั้น
รูปที่ 8.10 แสดงองค์ประกอบหลักของ OS
ที่เกี่ยวข้องกับมัลติโปรแกรมมิ่งและการจัดตารางเวลากระบวนการ

OS รับการควบคุมหน่วยประมวลผลที่ interrupt handler หากเกิดการขัดจังหวะ
และที่ service-call handler หากมีการเรียกบริการ
เมื่อการขัดจังหวะหรือการเรียกบริการถูกจัดการแล้ว
ตัวกำหนดตารางเวลาระยะสั้นจะถูกเรียกใช้เพื่อเลือกกระบวนการสำหรับการประมวลผล

เพื่อทำงาน OS ดูแลรักษาคิวหลายคิว
แต่ละคิวเป็นเพียงรายการรอของกระบวนการที่รอทรัพยากรบางอย่าง
long-term queue คือรายการงานที่รอใช้ระบบ
เมื่อสภาวะอนุญาต high-level scheduler จะจัดสรรหน่วยความจำ
และสร้างกระบวนการสำหรับรายการที่รออยู่หนึ่งรายการ
short-term queue ประกอบด้วยกระบวนการทั้งหมดในสถานะ ready
กระบวนการใดก็ตามในบรรดาเหล่านี้สามารถใช้หน่วยประมวลผลต่อไปได้
เป็นหน้าที่ของตัวกำหนดตารางเวลาระยะสั้นในการเลือกหนึ่งกระบวนการ
โดยทั่วไปนี้ทำด้วย round-robin algorithm
โดยให้เวลากับแต่ละกระบวนการตามลำดับ
อาจใช้ระดับความสำคัญด้วย
สุดท้าย มี I/O queue สำหรับอุปกรณ์อินพุต/เอาต์พุตแต่ละชนิด
มากกว่าหนึ่งกระบวนการอาจร้องขอการใช้อุปกรณ์อินพุต/เอาต์พุตเดียวกัน
กระบวนการทั้งหมดที่รอใช้อุปกรณ์แต่ละชนิดจะเรียงคิวในคิวของอุปกรณ์นั้น

293 บทที่ 8 / การสนับสนุนของระบบปฏิบัติการ

8.3 การจัดการหน่วยความจำ (Memory Management)

ในระบบยูนิโปรแกรมมิ่ง หน่วยความจำหลักถูกแบ่งออกเป็นสองส่วน
ส่วนหนึ่งสำหรับระบบปฏิบัติการ (resident monitor)
และอีกส่วนหนึ่งสำหรับโปรแกรมที่กำลังประมวลผลอยู่
ในระบบมัลติโปรแกรมมิ่ง ส่วน "ผู้ใช้" ของหน่วยความจำถูกแบ่งย่อย
เพื่อรองรับหลายกระบวนการ
งานการแบ่งย่อยนี้ดำเนินการแบบไดนามิกโดยระบบปฏิบัติการ
และเรียกว่าการจัดการหน่วยความจำ

การจัดการหน่วยความจำที่มีประสิทธิภาพมีความสำคัญในระบบมัลติโปรแกรมมิ่ง
หากมีกระบวนการอยู่ในหน่วยความจำเพียงไม่กี่กระบวนการ
กระบวนการทั้งหมดจะรออินพุต/เอาต์พุตส่วนใหญ่
และหน่วยประมวลผลจะว่างงาน
ดังนั้น หน่วยความจำต้องถูกจัดสรรอย่างมีประสิทธิภาพ
เพื่อบรรจุกระบวนการให้ได้มากที่สุดเท่าที่จะเป็นไปได้

การสวอปปิ้ง (Swapping)

กลับไปยังรูปที่ 8.11 เราได้กล่าวถึงคิวสามประเภท ได้แก่
long-term queue ของคำขอสำหรับกระบวนการใหม่
short-term queue ของกระบวนการที่พร้อมจะใช้หน่วยประมวลผล
และ I/O queue ต่าง ๆ ของกระบวนการที่ยังไม่พร้อมจะใช้หน่วยประมวลผล
เหตุผลของกลไกที่ซับซ้อนนี้คือ
กิจกรรมอินพุต/เอาต์พุตช้ากว่าการคำนวณมาก
และหน่วยประมวลผลในระบบยูนิโปรแกรมมิ่งจึงว่างงานส่วนใหญ่

แต่การจัดการในรูปที่ 8.11 ไม่ได้แก้ปัญหาได้ทั้งหมด
จริงอยู่ที่ในกรณีนี้ หน่วยความจำมีหลายกระบวนการ
และหน่วยประมวลผลสามารถย้ายไปยังกระบวนการอื่นได้เมื่อกระบวนการหนึ่งรอ
แต่หน่วยประมวลผลเร็วกว่าอินพุต/เอาต์พุตมาก
จนเป็นเรื่องปกติที่กระบวนการทั้งหมดในหน่วยความจำจะรออินพุต/เอาต์พุต
ดังนั้น แม้กระทั่งกับมัลติโปรแกรมมิ่ง
หน่วยประมวลผลก็อาจว่างงานส่วนใหญ่ของเวลา

ต้องทำอะไร? หน่วยความจำหลักสามารถขยายได้เพื่อรองรับกระบวนการได้มากขึ้น
แต่มีข้อบกพร่องสองประการในแนวทางนี้
ประการแรก หน่วยความจำหลักมีราคาแพง แม้กระทั่งในปัจจุบัน
ประการที่สอง ความต้องการหน่วยความจำของโปรแกรมเพิ่มขึ้นเร็ว
พอ ๆ กับที่ต้นทุนหน่วยความจำลดลง
ดังนั้นหน่วยความจำที่ใหญ่กว่าจึงให้กระบวนการที่ใหญ่กว่า ไม่ใช่กระบวนการที่มากกว่า

ทางออกอื่นคือการสวอปปิ้ง ดังที่แสดงในรูปที่ 8.12
มี long-term queue ของคำขอกระบวนการ ซึ่งมักเก็บบนดิสก์
กระบวนการเหล่านี้ถูกนำเข้ามาครั้งละหนึ่งเมื่อพื้นที่ว่าง
เมื่อกระบวนการเสร็จสมบูรณ์ พวกมันถูกย้ายออกจากหน่วยความจำหลัก
ตอนนี้สถานการณ์จะเกิดขึ้นที่ไม่มีกระบวนการใดในหน่วยความจำอยู่ในสถานะ ready
(เช่น ทั้งหมดรออินพุต/เอาต์พุต)
แทนที่จะว่างงาน หน่วยประมวลผลสวอปกระบวนการหนึ่งกลับออกไปยังดิสก์
ใน intermediate queue
นี่คือคิวของกระบวนการที่มีอยู่ซึ่งถูกขับออกจากหน่วยความจำชั่วคราว
จากนั้น OS นำกระบวนการอื่นมาจาก intermediate queue
หรือตอบสนองคำขอกระบวนการใหม่จาก long-term queue
การประมวลผลจะดำเนินต่อกับกระบวนการที่เพิ่งมาถึง

การสวอปปิ้งเป็นการดำเนินการอินพุต/เอาต์พุต
ดังนั้นจึงมีความเป็นไปได้ที่จะทำให้ปัญหาแย่ลง ไม่ใช่ดีขึ้น
แต่เนื่องจากดิสก์อินพุต/เอาต์พุตโดยทั่วไปเป็นอินพุต/เอาต์พุตที่เร็วที่สุดในระบบ
การสวอปปิ้งจึงมักปรับปรุงประสิทธิภาพ
รูปแบบที่ซับซ้อนกว่า ซึ่งเกี่ยวข้องกับหน่วยความจำเสมือน
จะปรับปรุงประสิทธิภาพเหนือการสวอปปิ้งอย่างง่าย
สิ่งนี้จะถูกกล่าวถึงในเร็ว ๆ นี้
แต่ก่อนอื่นเราต้องเตรียมพื้นฐานโดยอธิบายการแบ่งพาร์ติชันและเพจจิ้ง

การแบ่งพาร์ติชัน (Partitioning)

รูปแบบที่ง่ายที่สุดในการแบ่งพาร์ติชันหน่วยความจำที่มีอยู่
คือการใช้พาร์ติชันขนาดคงที่ (fixed-size partitions)
ดังที่แสดงในรูปที่ 8.13
โปรดสังเกตว่าแม้ว่าพาร์ติชันจะมีขนาดคงที่ แต่ก็ไม่จำเป็นต้องเท่ากัน
เมื่อกระบวนการถูกนำเข้าสู่หน่วยความจำ
มันจะถูกวางไว้ในพาร์ติชันที่เล็กที่สุดที่จะรองรับได้

แม้จะใช้พาร์ติชันขนาดคงที่ไม่เท่ากัน ก็จะมีหน่วยความจำที่สูญเปล่า
ในกรณีส่วนใหญ่ กระบวนการจะไม่ต้องการหน่วยความจำเท่ากับพาร์ติชันที่กำหนดพอดี
หน่วยความจำที่สูญเปล่าภายในพาร์ติชันนี้เรียกว่า internal fragmentation

ทางออกที่ดีกว่าคือการใช้พาร์ติชันขนาดแปรผัน (variable-size partitions)
ในกรณีนี้ หน่วยความจำจะถูกจัดสรรให้กับกระบวนการตามขนาดที่แน่นอนที่กระบวนการต้องการ
เมื่อกระบวนการออกจากหน่วยความจำ
หน่วยความจำที่ว่างจะกลับสู่ "pool" ของหน่วยความจำที่ว่าง
แม้ว่าเทคนิคนี้จะดีกว่า แต่มันมีความซับซ้อนในการจัดการ
ปัญหาอีกอย่างที่มักเกิดขึ้นกับการแบ่งพาร์ติชันแบบไดนามิกคือ external fragmentation
เมื่อเวลาผ่านไป หน่วยความจำจะมีหลุม (holes) เล็ก ๆ จำนวนมาก
ที่ไม่สามารถใช้ได้

เพื่อเอาชนะ external fragmentation
สามารถใช้เทคนิคที่เรียกว่าการบดอัด (compaction) ได้
ระบบปฏิบัติการจะย้ายกระบวนการในหน่วยความจำ
เพื่อให้กระบวนการต่อเนื่องกันและหน่วยความจำที่ว่างทั้งหมดอยู่ในบล็อกเดียว
อย่างไรก็ตาม การบดอัดเป็นกระบวนการที่ใช้เวลาและทรัพยากรมาก

เพจจิ้ง (Paging)

ทั้งพาร์ติชันขนาดคงที่และขนาดแปรผันใช้งานหน่วยความจำที่ต่อเนื่องกันสำหรับกระบวนการ
เพจจิ้งเป็นการปรับปรุงที่ทำให้เฟรมหน่วยความจำที่ไม่ต่อเนื่องกัน
สามารถใช้สำหรับกระบวนการได้

หน่วยความจำหลักถูกแบ่งออกเป็นชิ้นส่วนขนาดเล็กที่มีขนาดเท่ากันจำนวนมาก
เรียกว่าเฟรม (frames)
แต่ละกระบวนการถูกแบ่งออกเป็นเพจ (pages) ที่มีขนาดเท่ากับเฟรม
กระบวนการที่เล็กกว่าต้องการเพจน้อยกว่า
และกระบวนการที่ใหญ่กว่าต้องการเพจมากกว่า
เมื่อกระบวนการถูกนำเข้ามา เพจของมันจะถูกโหลดเข้าสู่เฟรมที่ว่าง
และตารางเพจจะถูกตั้งค่า

ในโปรแกรม ที่อยู่เชิงตรรกะแต่ละที่อยู่ประกอบด้วยหมายเลขเพจ
และที่อยู่สัมพัทธ์ภายในเพจ
ระบบปฏิบัติการดูแลรักษาตารางเพจ (page table) สำหรับแต่ละกระบวนการ
ตารางเพจแสดงตำแหน่งเฟรมสำหรับแต่ละเพจของกระบวนการ
หน่วยประมวลผลใช้ตารางเพจเพื่อสร้างที่อยู่ทางกายภาพ
เมื่อได้รับที่อยู่เชิงตรรกะ (หมายเลขเพจ, ที่อยู่สัมพัทธ์)
หน่วยประมวลผลใช้ตารางเพจเพื่อสร้างที่อยู่ทางกายภาพ
(หมายเลขเฟรม, ที่อยู่สัมพัทธ์)

วิธีนี้แก้ปัญหาที่กล่าวไว้ก่อนหน้า
หน่วยความจำหลักถูกแบ่งออกเป็นเฟรมขนาดเล็กที่มีขนาดเท่ากันจำนวนมาก
แต่ละกระบวนการถูกแบ่งออกเป็นเพจที่มีขนาดเท่ากับเฟรม
กระบวนการที่เล็กกว่าต้องการเพจน้อยกว่า
กระบวนการที่ใหญ่กว่าต้องการเพจมากกว่า
เมื่อกระบวนการถูกนำเข้ามา เพจของมันจะถูกโหลดเข้าสู่เฟรมที่ว่าง
และตารางเพจถูกตั้งค่า

หน่วยความจำเสมือน (Virtual Memory)

ด้วยการใช้เพจจิ้ง ระบบมัลติโปรแกรมมิ่งที่มีประสิทธิภาพอย่างแท้จริงได้เกิดขึ้น
นอกจากนี้ กลวิธีง่าย ๆ ของการแบ่งกระบวนการออกเป็นเพจ
นำไปสู่การพัฒนาแนวคิดสำคัญอีกอย่างหนึ่ง ได้แก่ หน่วยความจำเสมือน

เพื่อให้เข้าใจหน่วยความจำเสมือน เราต้องเพิ่มการปรับปรุงในรูปแบบเพจจิ้งที่กล่าวถึง
การปรับปรุงนั้นคือ demand paging
ซึ่งหมายความว่าแต่ละเพจของกระบวนการจะถูกนำเข้ามาเมื่อต้องการเท่านั้น
กล่าวคือ ตามความต้องการ

ลองนึกถึงกระบวนการขนาดใหญ่
ซึ่งประกอบด้วยโปรแกรมยาวบวกกับอาร์เรย์ข้อมูลหลายชุด
ในช่วงเวลาสั้น ๆ การประมวลผลอาจจำกัดอยู่ในส่วนเล็ก ๆ ของโปรแกรม
(เช่น ซับรูทีน)
และอาจใช้เพียงหนึ่งหรือสองอาร์เรย์ข้อมูล
นี่คือหลักการของพื้นที่ (principle of locality)
ซึ่งเราได้แนะนำไว้ในภาคผนวก 4A
การโหลดเพจหลายสิบเพจสำหรับกระบวนการนั้น
เมื่อจะใช้เพียงไม่กี่เพจก่อนที่โปรแกรมจะถูกระงับนั้นเป็นสิ่งที่สูญเปล่าอย่างชัดเจน
เราสามารถใช้หน่วยความจำได้ดีขึ้นโดยการโหลดเพียงไม่กี่เพจ
จากนั้นหากโปรแกรม branch ไปยังคำสั่งในเพจที่ไม่ได้อยู่ในหน่วยความจำหลัก
หรือโปรแกรมอ้างอิงข้อมูลในเพจที่ไม่ได้อยู่ในหน่วยความจำ จะเกิด page fault
สิ่งนี้บอกให้ระบบปฏิบัติการนำเพจที่ต้องการเข้ามา

ดังนั้น ณ จุดเวลาใดก็ตาม มีเพียงไม่กี่เพจของกระบวนการที่กำหนดอยู่ในหน่วยความจำ
และกระบวนการจำนวนมากขึ้นสามารถอยู่ในหน่วยความจำได้
นอกจากนี้ยังประหยัดเวลาเพราะเพจที่ไม่ได้ใช้ไม่ถูกสวอปเข้าออกจากหน่วยความจำ

อย่างไรก็ตาม ระบบปฏิบัติการต้องฉลาดเกี่ยวกับวิธีจัดการรูปแบบนี้
เมื่อนำเพจหนึ่งเข้ามา ต้องนำเพจอื่นออก ซึ่งรู้จักกันในชื่อ page replacement
หากนำเพจออกก่อนที่มันจะถูกใช้
จะต้องนำมันกลับมาใหม่เกือบทันที
การทำเช่นนี้มากเกินไปนำไปสู่สภาวะที่เรียกว่า thrashing
ที่หน่วยประมวลผลใช้เวลาส่วนใหญ่ในการสวอปเพจ
แทนที่จะประมวลผลคำสั่ง
การหลีกเลี่ยง thrashing เป็นพื้นที่การวิจัยสำคัญในทศวรรษ 1970
และนำไปสู่อัลกอริทึมที่ซับซ้อนแต่มีประสิทธิภาพหลากหลาย
โดยพื้นฐานแล้ว ระบบปฏิบัติการพยายามเดาโดยอิงจากประวัติล่าสุด
ว่าเพจใดน่าจะถูกใช้น้อยที่สุดในอนาคตอันใกล้

ด้วย demand paging ไม่จำเป็นต้องโหลดกระบวนการทั้งหมดเข้าสู่หน่วยความจำหลัก
ข้อเท็จจริงนี้มีผลสืบเนื่องที่น่าทึ่ง
คือ กระบวนการสามารถมีขนาดใหญ่กว่าหน่วยความจำหลักทั้งหมดได้
ข้อจำกัดพื้นฐานที่สุดอย่างหนึ่งในการโปรแกรมถูกยกออกไป
หากไม่มี demand paging โปรแกรมเมอร์ต้องตระหนักอย่างชัดเจนว่ามีหน่วยความจำเพียงใด
หากโปรแกรมที่กำลังเขียนมีขนาดใหญ่เกินไป
โปรแกรมเมอร์ต้องคิดวิธีจัดโครงสร้างโปรแกรมเป็นชิ้นส่วน
ที่สามารถโหลดได้ครั้งละหนึ่งชิ้น
ด้วย demand paging งานนั้นเป็นหน้าที่ของระบบปฏิบัติการและฮาร์ดแวร์
สำหรับโปรแกรมเมอร์ เขาหรือเธอกำลังจัดการกับหน่วยความจำขนาดใหญ่มาก
ซึ่งมีขนาดเกี่ยวข้องกับการจัดเก็บดิสก์

เนื่องจากกระบวนการทำงานในหน่วยความจำหลักเท่านั้น
หน่วยความจำนั้นจึงเรียกว่า real memory
แต่โปรแกรมเมอร์หรือผู้ใช้รับรู้หน่วยความจำที่ใหญ่กว่ามาก
นั่นคือที่ถูกจัดสรรบนดิสก์
ดังนั้นหน่วยความจำหลังนี้จึงเรียกว่า virtual memory
หน่วยความจำเสมือนช่วยให้มัลติโปรแกรมมิ่งมีประสิทธิภาพสูงมาก
และปลดปล่อยผู้ใช้จากข้อจำกัดอันแน่นหนาของหน่วยความจำหลัก

โครงสร้างตารางเพจ (Page Table Structure)

กลไกพื้นฐานสำหรับการอ่านคำจากหน่วยความจำ
เกี่ยวข้องกับการแปลที่อยู่เสมือนหรือเชิงตรรกะ
ซึ่งประกอบด้วยหมายเลขเพจและออฟเซ็ต
ไปเป็นที่อยู่ทางกายภาพ ซึ่งประกอบด้วยหมายเลขเฟรมและออฟเซ็ต
โดยใช้ตารางเพจ

ในระบบส่วนใหญ่ มีตารางเพจหนึ่งตารางต่อกระบวนการ
แต่แต่ละกระบวนการอาจครอบครองหน่วยความจำเสมือนจำนวนมาก
ในระบบส่วนใหญ่ มีการจัดเก็บตารางเพจในหน่วยความจำเสมือน
แทนที่จะเป็นหน่วยความจำจริง
ซึ่งหมายความว่าตารางเพจสามารถถูกเพจได้เช่นเดียวกับเพจอื่น ๆ
เมื่อกระบวนการทำงาน
อย่างน้อยส่วนหนึ่งของตารางเพจต้องอยู่ในหน่วยความจำหลัก
รวมถึงรายการตารางเพจของเพจที่กำลังประมวลผลอยู่

โปรเซสเซอร์บางตัวใช้รูปแบบสองระดับเพื่อจัดระเบียบตารางเพจขนาดใหญ่
ในรูปแบบนี้ มี page directory ที่แต่ละรายการชี้ไปยังตารางเพจ
หากความยาวของ page directory คือ X
และหากความยาวสูงสุดของตารางเพจคือ Y
กระบวนการสามารถประกอบด้วยเพจได้สูงสุด X × Y เพจ

แนวทางอื่นสำหรับการใช้ตารางเพจหนึ่งหรือสองระดับ
คือการใช้โครงสร้าง inverted page table (รูปที่ 8.17)
ในแนวทางนี้ หมายเลขเพจของที่อยู่เสมือน
ถูกแมปไปยังค่าแฮชโดยใช้ฟังก์ชันแฮชอย่างง่าย
ค่าแฮชเป็นตัวชี้ไปยัง inverted page table ซึ่งมีรายการตารางเพจ
มีรายการหนึ่งใน inverted page table สำหรับแต่ละเฟรมหน่วยความจำจริง
แทนที่จะเป็นหนึ่งต่อเพจเสมือน
ดังนั้นจึงต้องใช้หน่วยความจำจริงในสัดส่วนคงที่สำหรับตาราง
โดยไม่คำนึงถึงจำนวนกระบวนการหรือเพจเสมือนที่รองรับ

Translation Lookaside Buffer (TLB)

ในหลักการแล้ว การอ้างอิงหน่วยความจำเสมือนทุกครั้ง
อาจทำให้เกิดการเข้าถึงหน่วยความจำทางกายภาพสองครั้ง
ครั้งหนึ่งเพื่อดึงรายการตารางเพจที่เหมาะสม
และอีกครั้งเพื่อดึงข้อมูลที่ต้องการ
ดังนั้นรูปแบบหน่วยความจำเสมือนแบบตรงไปตรงมา
จะมีผลทำให้เวลาเข้าถึงหน่วยความจำเพิ่มขึ้นเป็นสองเท่า

เพื่อแก้ปัญหานี้ รูปแบบหน่วยความจำเสมือนส่วนใหญ่
ใช้แคชพิเศษสำหรับรายการตารางเพจ
มักเรียกว่า Translation Lookaside Buffer (TLB)
แคชนี้ทำงานในลักษณะเดียวกับแคชหน่วยความจำ
และมีรายการตารางเพจที่ถูกใช้ล่าสุด
รูปที่ 8.18 เป็นผังงานที่แสดงการใช้ TLB

โดยหลักการของพื้นที่
การอ้างอิงหน่วยความจำเสมือนส่วนใหญ่
จะเกี่ยวกับตำแหน่งในเพจที่ถูกใช้ล่าสุด
ดังนั้น การอ้างอิงส่วนใหญ่จะเกี่ยวข้องกับรายการตารางเพจใน TLB

โปรดสังเกตว่ากลไกหน่วยความจำเสมือนต้องโต้ตอบกับระบบแคช
(ไม่ใช่แคช TLB แต่เป็นแคชหน่วยความจำหลัก)
ที่อยู่เสมือนโดยทั่วไปจะอยู่ในรูปแบบของหมายเลขเพจ, ออฟเซ็ต
ขั้นแรก ระบบหน่วยความจำปรึกษา TLB
เพื่อดูว่ารายการตารางเพจที่ตรงกันมีอยู่หรือไม่
หากมี ที่อยู่จริง (ทางกายภาพ) จะถูกสร้างขึ้น
โดยการรวมหมายเลขเฟรมกับออฟเซ็ต
หากไม่มี รายการจะถูกเข้าถึงจากตารางเพจ
เมื่อสร้างที่อยู่จริงแล้ว แคชจะถูกปรึกษา
เพื่อดูว่าบล็อกที่มีคำนั้นมีอยู่หรือไม่
หากมี มันจะถูกส่งคืนไปยังหน่วยประมวลผล
หากไม่มี คำนั้นจะถูกดึงจากหน่วยความจำหลัก

เซกเมนเตชัน (Segmentation)

เซกเมนเตชันช่วยให้โปรแกรมเมอร์มองหน่วยความจำว่า
ประกอบด้วยเซกเมนต์หลายส่วน แต่ละส่วนมีขนาดแตกต่างกัน
ในแนวทางนี้ โปรแกรมและข้อมูลที่เกี่ยวข้อง
อาจถูกแบ่งออกเป็นเซกเมนต์ที่ผู้โปรแกรมเมอร์กำหนดขึ้น
ที่อยู่ตรรกะประกอบด้วยหมายเลขเซกเมนต์และออฟเซ็ต

เซกเมนเตชันคล้ายกับการแบ่งพาร์ติชันแบบไดนามิก
ยกเว้นว่าโปรแกรมในเซกเมนเตชันอาจครอบครองมากกว่าหนึ่งพาร์ติชัน
และพาร์ติชันเหล่านั้นไม่จำเป็นต้องต่อเนื่องกัน

เซกเมนเตชันขจัด internal fragmentation
แต่เช่นเดียวกับการแบ่งพาร์ติชันแบบไดนามิก
มันต้องทนกับ external fragmentation
อย่างไรก็ตาม เนื่องจากกระบวนการถูกแบ่งออกเป็นชิ้นส่วนหลายชิ้นที่ประกอบกัน
การเคลื่อนไหวที่ต้องการสำหรับการบดอัดจึงน้อยกว่า

8.4 การจัดการหน่วยความจำของ Intel x86

พื้นที่ที่อยู่ (Address Spaces)

โปรเซสเซอร์ x86 รองรับการใช้ที่อยู่ในสามระดับที่แตกต่างกัน

ในโปรแกรมที่ใช้เซกเมนเตชัน ที่อยู่เชิงตรรกะหรือ virtual address
ประกอบด้วย segment number และ offset ภายในเซกเมนต์
โดยเซกเมนเตชัน hardware แปลงนี้ไปเป็น linear address ของ 32 บิต
จากนั้น paging hardware แปลง linear address
ให้เป็น physical address ใน 32 บิต

เซกเมนเตชัน (Segmentation)

ใน x86 ที่อยู่เชิงตรรกะประกอบด้วยสองส่วนคือ

■ Segment selector: ตัวระบุ 16 บิตที่ประกอบด้วย
  Index: ดัชนีในตารางเซกเมนต์ (segment table)
  Requested Privilege Level (RPL): ระดับสิทธิ์ที่ร้องขอสำหรับการเข้าถึงนี้

แต่ละรายการในตารางเซกเมนต์ประกอบด้วย 64 บิต
ดังแสดงในรูปที่ 8.20c

เพจจิ้ง (Paging)

เซกเมนเตชันเป็นคุณสมบัติเพิ่มเติมและอาจถูกปิดใช้งานได้
เมื่อเซกเมนเตชันถูกใช้งาน ที่อยู่ที่ใช้ในโปรแกรมคือ virtual address
และถูกแปลงเป็น linear address ดังที่อธิบายไว้
เมื่อไม่ใช้เซกเมนเตชัน จะใช้ linear address ในโปรแกรม
ในกรณีใดก็ตาม ขั้นตอนต่อไปคือการแปลง linear address นั้น
เป็น real 32-bit address

กลไกเพจจิ้งของ x86 จริง ๆ แล้วเป็นการค้นหาตารางสองระดับ
ระดับแรกคือ page directory ซึ่งมีรายการได้สูงสุด 1,024 รายการ
ซึ่งแบ่งพื้นที่หน่วยความจำ linear ขนาด 4 GB
ออกเป็น 1,024 กลุ่มเพจ
แต่ละกลุ่มมี page table ของตัวเอง และแต่ละกลุ่มยาว 4 MB
แต่ละ page table มีรายการได้สูงสุด 1,024 รายการ
แต่ละรายการตรงกับเพจขนาด 4 KB หนึ่งเพจ

การจัดการหน่วยความจำมีตัวเลือกในการใช้ page directory หนึ่งตารางสำหรับกระบวนการทั้งหมด
หนึ่งตารางสำหรับแต่ละกระบวนการ
หรือการรวมกันของทั้งสอง
page directory สำหรับงานปัจจุบันจะอยู่ในหน่วยความจำหลักเสมอ
Page table อาจอยู่ใน virtual memory

x86 ยังใช้ TLB ที่สามารถเก็บรายการตารางเพจได้ 32 รายการ
ทุกครั้งที่ page directory เปลี่ยนแปลง บัฟเฟอร์จะถูกล้างออก

รูปที่ 8.21 แสดงการรวมกันของกลไกเซกเมนเตชันและเพจจิ้ง

สุดท้าย x86 รวมส่วนขยายใหม่ที่ไม่พบใน 80386 หรือ 80486 รุ่นก่อนหน้า
คือการรองรับสองขนาดเพจ
หากบิต PSE (page size extension) ในตัวควบคุมรีจิสเตอร์ 4 ถูกตั้งค่าเป็น 1
หน่วยเพจจิ้งจะอนุญาตให้โปรแกรมเมอร์ OS กำหนดเพจ
ให้มีขนาด 4 KB หรือ 4 MB

เมื่อใช้เพจขนาด 4 MB
จะมีการค้นหาตารางเพียงระดับเดียวสำหรับเพจ
เมื่อฮาร์ดแวร์เข้าถึง page directory
รายการ page directory (รูปที่ 8.20d) จะมีบิต PS ถูกตั้งค่าเป็น 1
ในกรณีนี้ bits 9 ถึง 21 จะถูกละเว้น
และ bits 22 ถึง 31 กำหนดที่อยู่ฐานสำหรับเพจขนาด 4 MB ในหน่วยความจำ
ดังนั้นจึงมี page table เดียว

การใช้เพจขนาด 4 MB ลดข้อกำหนดการจัดเก็บการจัดการหน่วยความจำ
สำหรับหน่วยความจำหลักขนาดใหญ่
ด้วยเพจขนาด 4 KB หน่วยความจำหลักขนาด 4 GB เต็ม
ต้องการหน่วยความจำประมาณ 4 MB เพียงสำหรับ page table
ด้วยเพจขนาด 4 MB ตารางเดียวขนาด 4 KB
เพียงพอสำหรับการจัดการหน่วยความจำเพจ

8.5 การจัดการหน่วยความจำของ ARM (ARM Memory Management)

ARM มอบสถาปัตยกรรมระบบหน่วยความจำเสมือนที่หลากหลาย
ซึ่งสามารถปรับแต่งให้ตรงกับความต้องการของนักออกแบบระบบฝังตัว

การจัดระบบหน่วยความจำ (Memory System Organization)

รูปที่ 8.22 ให้ภาพรวมของฮาร์ดแวร์การจัดการหน่วยความจำใน ARM
สำหรับหน่วยความจำเสมือน
ฮาร์ดแวร์แปลหน่วยความจำเสมือนใช้ตารางหนึ่งหรือสองระดับ
สำหรับการแปลจากที่อยู่เสมือนเป็นที่อยู่ทางกายภาพ
Translation lookaside buffer (TLB) คือแคชของรายการตารางเพจล่าสุด
หากรายการมีอยู่ใน TLB
TLB จะส่งที่อยู่ทางกายภาพไปยังหน่วยความจำหลักโดยตรง
สำหรับการดำเนินการอ่านหรือเขียน
ข้อมูลถูกแลกเปลี่ยนระหว่างหน่วยประมวลผลและหน่วยความจำหลักผ่านแคช

รายการในตารางแปลยังรวมถึง access control bits
ซึ่งกำหนดว่ากระบวนการที่กำหนดอาจเข้าถึงส่วนที่กำหนดของหน่วยความจำได้หรือไม่
หากการเข้าถึงถูกปฏิเสธ
ฮาร์ดแวร์ควบคุมการเข้าถึงจะส่งสัญญาณ abort ไปยังโปรเซสเซอร์ ARM

การแปลที่อยู่หน่วยความจำเสมือน (Virtual Memory Address Translation)

ARM รองรับการเข้าถึงหน่วยความจำโดยอิงตาม sections หรือ pages

■ Section: บล็อกขนาด 1 MB ของหน่วยความจำทางกายภาพ
■ Large page: 64 KB
■ Small page: 4 KB
■ Tiny page: 1 KB

การแปลที่อยู่โดยอิง section
ต้องการการค้นหาตารางเพียงระดับเดียว
การแปลโดยอิง page ต้องการสองระดับ

รูปแบบการจัดการหน่วยความจำ (Memory-Management Formats)

ARM ใช้ first-level descriptor table
ซึ่งมีรายการ 4,096 รายการ (ดัชนีโดย 12 บิตบน)
แต่ละรายการอธิบาย section ขนาด 1 MB
หรือชี้ไปยัง second-level table สำหรับการแปลแบบเพจ

การควบคุมการเข้าถึง (Access Control)

ARM ใช้ระบบ domain สำหรับการควบคุมการเข้าถึง
แต่ละ section หรือเพจถูกกำหนดให้เป็นส่วนหนึ่งของ domain
Domain Access Control Register กำหนดว่า domain ใด
ที่โปรแกรมที่กำลังทำงานอยู่สามารถเข้าถึงได้

8.6 คำศัพท์สำคัญ คำถามทบทวน และปัญหา

คำศัพท์สำคัญ

demand paging: เทคนิคที่เพจของกระบวนการถูกโหลดเข้าสู่หน่วยความจำหลักตามที่ต้องการเท่านั้น
dispatcher: ตัวกำหนดตารางเวลาระยะสั้นที่ตัดสินใจว่ากระบวนการใดจะถูกประมวลผลต่อไป
external fragmentation: พื้นที่หน่วยความจำที่ถูกแบ่งเป็นชิ้นส่วนเล็ก ๆ และไม่ได้ใช้ประโยชน์
frame: บล็อกขนาดเล็กของหน่วยความจำหลักในระบบเพจจิ้ง
internal fragmentation: การสูญเสียพื้นที่ภายในพาร์ติชันหน่วยความจำที่จัดสรรให้กับกระบวนการ
interrupt: สัญญาณที่สร้างโดยฮาร์ดแวร์เพื่อบอกให้ระบบปฏิบัติการทราบว่ามีเหตุการณ์ที่ต้องการความสนใจ
job control language (JCL): ภาษาพิเศษที่ใช้เพื่อให้คำสั่งแก่ monitor ในระบบแบตช์
kernel: ส่วนหลักของระบบปฏิบัติการที่อยู่ในหน่วยความจำหลักเสมอ
long-term scheduling: การตัดสินใจว่าจะรับกระบวนการใหม่เข้าสู่ระบบหรือไม่
medium-term scheduling: การตัดสินใจเกี่ยวกับการสวอปกระบวนการเข้าและออกจากหน่วยความจำหลัก
memory management: ฟังก์ชันของระบบปฏิบัติการที่จัดสรรหน่วยความจำสำหรับกระบวนการ
multiprogramming: เทคนิคที่โปรแกรมหลายโปรแกรมอยู่ในหน่วยความจำพร้อมกัน
operating system (OS): โปรแกรมที่ควบคุมการทำงานของโปรแกรมประยุกต์
page: หน่วยของข้อมูลในระบบเพจจิ้ง
page fault: เหตุการณ์ที่เกิดขึ้นเมื่อกระบวนการอ้างอิงเพจที่ไม่ได้อยู่ในหน่วยความจำหลัก
page replacement: กระบวนการของการนำเพจออกจากหน่วยความจำหลักเพื่อให้มีพื้นที่สำหรับเพจอื่น
page table: ตารางที่ใช้โดยระบบปฏิบัติการเพื่อแปลที่อยู่เพจเสมือนเป็นที่อยู่เฟรมจริง
paging: เทคนิคการจัดการหน่วยความจำที่กระบวนการถูกแบ่งออกเป็นเพจขนาดคงที่
partitioning: การแบ่งหน่วยความจำหลักออกเป็นส่วนสำหรับกระบวนการ
process: โปรแกรมที่กำลังทำงานพร้อมกับข้อมูลและบริบท
process control block (PCB): โครงสร้างข้อมูลที่เก็บข้อมูลเกี่ยวกับกระบวนการ
process state: สถานะปัจจุบันของกระบวนการ (new, ready, running, waiting, halted)
real memory: หน่วยความจำหลักทางกายภาพที่กระบวนการทำงานอยู่
resident monitor: ส่วนของ monitor ที่อยู่ในหน่วยความจำหลักเสมอ
scheduling: ฟังก์ชันของระบบปฏิบัติการที่กำหนดลำดับที่โปรแกรมรับทรัพยากร
segment: ส่วนของโปรแกรมในระบบเซกเมนเตชัน
segmentation: เทคนิคการจัดการหน่วยความจำที่โปรแกรมถูกแบ่งออกเป็นส่วนที่มีขนาดแตกต่างกัน
short-term scheduling: การตัดสินใจว่าจะประมวลผลกระบวนการใดต่อไป
swapping: เทคนิคการจัดการหน่วยความจำที่กระบวนการถูกย้ายชั่วคราวจากหน่วยความจำหลักไปยังดิสก์
thrashing: สภาวะที่ระบบใช้เวลาส่วนใหญ่ในการสวอปเพจแทนที่จะประมวลผลคำสั่ง
translation lookaside buffer (TLB): แคชฮาร์ดแวร์ที่เก็บรายการตารางเพจล่าสุด
virtual memory: เทคนิคที่อนุญาตให้กระบวนการมีขนาดใหญ่กว่าหน่วยความจำหลักที่มีอยู่

คำถามทบทวน

8.1 ระบุและอธิบายวัตถุประสงค์หลักสองประการของระบบปฏิบัติการ
8.2 ระบุและอธิบายบริการหลักที่ระบบปฏิบัติการให้แก่ผู้ใช้
8.3 อธิบายความแตกต่างระหว่าง kernel และระบบปฏิบัติการ
8.4 อธิบายความแตกต่างระหว่างระบบ batch และ interactive
8.5 อธิบายว่า multiprogramming คืออะไรและทำไมจึงสำคัญ
8.6 อธิบายความแตกต่างระหว่าง long-term, medium-term และ short-term scheduling
8.7 อธิบาย process state ห้าสถานะที่กล่าวถึงในบทนี้
8.8 process control block คืออะไรและมีข้อมูลอะไรบ้าง
8.9 อธิบายความแตกต่างระหว่าง internal และ external fragmentation
8.10 อธิบายว่า demand paging คืออะไรและทำงานอย่างไร
8.11 translation lookaside buffer คืออะไรและทำไมจึงสำคัญ
8.12 อธิบายความแตกต่างระหว่าง paging และ segmentation
8.13 virtual memory คืออะไรและให้ประโยชน์อะไร
8.14 อธิบายกลไก page replacement และทำไม thrashing จึงเป็นปัญหา
8.15 กลไก TLB ในสถาปัตยกรรม x86 ทำงานอย่างไร
8.16 อธิบายวิธีที่ ARM จัดการ access control โดยใช้ domain
